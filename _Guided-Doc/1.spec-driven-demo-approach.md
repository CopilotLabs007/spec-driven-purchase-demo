# Spec-Driven Development Demo: Purchase Request API

## Goal
Show how a markdown specification drives API generation, changes, and validation.

## Demo App - Idea
Build a simple Purchase Request Tracker for a generic customer-friendly business scenario.

Core fields:
- id
- title
- requester
- amount
- costCenter
- status

Status values:
- draft
- submitted
- approved
- rejected

## Step-by-Step Flow

1. Create the specification.

```text
Use spec.md as the source of truth.
```

2. Use seed data.

```text
Load initial demo data from data/purchase-requests.seed.json.
```

3. Generate the API.

```text
Generate a Node.js Express API based on spec.md.
Add routes, controllers, models, validation, and in-memory storage.
```

4. Run the API.

```bash
npm install
npm start
```

5. Test the seeded API.

```bash
curl -i http://localhost:3000/purchase-requests
```

6. Update the specification.

```text
Update spec.md.
Add endpoint: GET /purchase-requests/status/{status}.
Return only purchase requests matching the status path parameter.
```

7. Update the implementation.

```text
Update the Node.js Express API to match the revised specification.
Add the new status filter endpoint.
```

8. Re-test the change.

```bash
curl -i http://localhost:3000/purchase-requests/status/submitted
```

## Demo Files
- spec.md
- specs/purchase-request.spec.md
- data/purchase-requests.seed.json

## Key Message
The specification is the source of truth. Change the spec first, then update code and tests from it.
