# 🏗️ Brownfield Spec-Driven Development

## 🎯 Goal

Use spec-driven development with an existing application by first discovering what the app already does, then turning that into a validated baseline specification.

## 🧭 Big Idea

In a greenfield project, the spec comes first.

In a brownfield project, the existing app comes first.

The goal is to move from undocumented behavior to a trusted spec-driven workflow.

## 🔍 Brownfield Demo Flow

1. 🔎 **Inspect the existing app**

   Review existing Express routes, controllers, models, validation logic, seed data, tests, and sample API calls.

2. 🧾 **Generate a baseline spec**

   Create an initial API specification from the current implementation.

3. ✅ **Validate the generated spec**

   Use curl, Postman, or integration tests to confirm the spec matches actual runtime behavior.

4. 🕵️ **Identify drift**

   Find undocumented behavior, missing validation, legacy fields, inconsistent responses, or routes used by clients but not documented.

5. ✏️ **Update the spec first**

   Add the desired change to the spec before touching implementation code.

6. 🛠️ **Update the implementation**

   Modify the existing Node.js Express API to match the revised specification.

7. 🧪 **Add regression tests**

   Protect existing behavior so the new change does not accidentally break current clients.

## 🔁 Workflow Comparison

```text
Greenfield:
spec -> code
```

```text
Brownfield:
code -> baseline spec -> validated spec -> changed spec -> code
```

## 📌 Demo Message

Spec-driven development still works for brownfield projects, but the starting point changes.

The existing application is the first source of truth. The spec becomes the source of truth after it is generated, reviewed, validated, and corrected.

## 🧩 Where SpecStory Helps

SpecStory gives the audience a timeline of the journey:

- 💬 What prompt started the change
- 📄 How the baseline spec was created
- 🔍 What behavior was discovered
- ✏️ What changed in the spec
- 🛠️ What changed in the implementation
- ✅ How the final behavior was validated

## ⭐ Key Takeaway

For brownfield projects, do not jump straight from idea to code.

First discover the current behavior, turn it into a baseline spec, validate it, and then make changes spec-first.
